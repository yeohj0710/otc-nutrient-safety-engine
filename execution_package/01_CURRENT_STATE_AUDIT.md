# 기존 산출물 감사 결과

## 1. 종합 판정

기존 작업은 프로토타입과 자료 정리의 출발점으로는 쓸 수 있지만, 현재 상태 그대로는 졸업논문의 연구 결과로 방어하기 어렵다. 가장 큰 문제는 문장 표현이 아니라 연구 연쇄의 단절이다. 검색된 전체 기록, 실제 선별 기록, 원문 검토, 비뚤림 위험 평가, 근거 합성, 규칙 생성, 독립 검증이 하나의 재현 가능한 경로로 이어지지 않는다.

현재 논문은 스스로 원문 검토와 접근 제한 데이터베이스 검색이 후속 과제라고 밝힌다. 이는 완성된 결과 논문이 아니라 진행 보고서에 가까운 상태임을 뜻한다. 핵심 본문도 방법·결과보다 일반 설명과 반복 부록이 많고, 오탈자와 번역 혼입이 있어 최종본으로 사용할 수 없다.

## 2. 치명적 불일치

### 2.1 학생·주제·저장소 불일치

- 기준 연구계획서는 여형준의 항응고제·신장 고위험군 연구다.
- 첨부 ZIP은 `권혁찬.zip`이다.
- 사용자가 제시한 `otc-nutrient-safety-engine` 공개 배포에는 권혁찬의 일반의약품형 고함량 영양성분 연구가 표시된다.
- 여형준 연구와 연결된 기존 프로젝트는 별도의 `nutrition-safety-engine`이다.

**조치:** Gate 0에서 저장소 원격 주소, 프로젝트명, 학생명, 학번, 연구 제목을 검사한다. 불일치하면 기존 저장소를 직접 덮어쓰지 않고 여형준 전용 브랜치·작업트리·저장소를 만든다. 권혁찬 자료는 범용 코드의 참고 자료로만 분리한다.

## 3. 검색과 선별의 문제

### 3.1 전체 검색 결과를 선별하지 않음

기존 검색 기록에는 PubMed 검색 결과가 7,243건과 1,714건으로 기록되어 있지만, 각 검색에서 관련도 순 상위 100건만 요청했다. 초기 검색도 상위 20건만 저장했다. 따라서 8,957건을 검색한 뒤 196건을 체계적으로 선별한 것이 아니라, 검색 엔진이 정렬한 일부만 가져온 것이다.

이 자료로는 PRISMA 흐름도를 만들 수 없다. `hit_count`는 검색식이 반환한 전체 개수이고, `stored_records`는 임의 상한으로 잘라 저장한 개수라서 같은 단계의 분모와 분자가 아니다.

### 3.2 보조 검색량의 과대 해석 위험

Europe PMC와 Crossref 검색은 각 질의의 상위 20건만 저장하면서 수십만 건의 전체 hit 수를 표시한다. Crossref의 광범위한 서지 검색 결과는 체계적 검색의 깊이 또는 선별 규모를 뜻하지 않는다. 보조 검색은 누락 대조 또는 DOI 보완으로만 사용해야 한다.

### 3.3 자동 분류의 거짓 양성

우선검토후보 파일에는 보충제 노출을 다루지 않는 `J-RISK AF Study`가 `EPA` 성분 일치로 포함 후보가 된 사례가 있다. 이는 짧은 문자열·문맥 미확인 매칭이 임상 노출로 오인될 수 있음을 보여준다. 기존 분류 결과의 90% 이상이 사람 검토 후보로 남아 있어 분류기가 실제 부담을 거의 줄이지 못했다.

### 3.4 연도 분포 오류

문헌특성분포 파일은 2020–2026년 문헌을 0건, 연도 누락을 236건으로 기록하지만 후보 파일에는 2024년 등 연도가 존재한다. 데이터 결합 또는 필드 매핑 오류다. 이 표와 이를 인용한 모든 문장은 폐기하고 원자료에서 다시 계산해야 한다.

## 4. 근거평가의 문제

- 핵심 근거 추출표는 10건뿐이며 체계적인 전문 검토 로그에서 도출되지 않았다.
- 연구설계에 따라 RoB 2, ROBINS-E/ROBINS-I, AMSTAR 2 등을 적용한 기록이 없다.
- `level 1/2/3` 같은 자체 등급은 기준이 불명확하고, 연구별 비뚤림 위험과 결과별 근거 확실성을 구분하지 않는다.
- 정부 fact sheet, RCT, 관찰연구, 사례보고가 서로 다른 역할임에도 하나의 근거 수준 표에 섞여 있다.
- 규칙 검증표의 상태가 `seed_requires_human_source_check`인 채로 앱 규칙에 연결되었다. 이는 근거 확인 전 단계다.

## 5. 시스템 검증의 문제

기존 시나리오 평가는 이미 작성한 규칙에 맞는 입력을 넣어 같은 규칙이 반환되는지 확인한다. 이는 회귀 테스트로는 유용하지만 독립적인 정확성 검증이 아니다. 규칙 작성에 사용하지 않은 사례, 음성 대조군, 경계 용량, 정보 누락, 충돌 근거를 포함한 외부 테스트셋이 필요하다.

또한 현재 엔진은 여형준 연구 범위를 넘어 갑상선호르몬·퀴놀론·임신·밀크시슬 등 많은 예시를 포함한다. 연구 질문과 앱 범위가 일치하지 않아 논문 결과의 분모와 시스템 기능의 분모가 달라진다.

## 6. 논문 문체와 구성의 문제

- 결과가 충분히 생성되지 않은 상태에서 결론형 문장이 먼저 작성되었다.
- 같은 설명을 부록에서 반복해 분량을 늘렸고, 핵심 분석은 짧다.
- `구성하였다`, `정리하였다`, `체계적으로`, `근거 기반` 같은 추상어가 반복된다.
- “상태을” 같은 오탈자, 영문 초록의 한국어 혼입, 문법 오류가 있다.
- 한 문단에 근거·방법·해석이 뒤섞여 독자가 실제로 무엇을 발견했는지 파악하기 어렵다.

## 7. 재사용 가능한 부분

- 항응고제와 신장결석 위험군이라는 임상 문제의식
- 비타민 K, 오메가-3, 칼슘, 비타민 D, 비타민 C의 5개 핵심 노드
- Next.js 기반 화면과 규칙 엔진의 범용 구조
- 검색 실행·원시 응답·규칙 추적표를 남기려 한 시도
- “진단·처방이 아니라 상담 준비 지원”이라는 제품 경계

## 8. 폐기 또는 격리할 부분

- 기존 PRISMA 수치와 흐름도
- 전체 검색량을 연구 규모처럼 표현한 문장
- 상위 N개 API 결과를 바탕으로 한 후보 비율
- 원문 확인 없는 rule seed
- 기존 자체 근거등급
- 독립 검증으로 표현된 시나리오 회귀 테스트
- 권혁찬 연구의 학생 정보·규칙·수치·문헌을 여형준 연구에 혼합한 부분

## 9. 감사 결론

기존 결과물을 “수정”하는 접근은 오히려 오류를 보존할 가능성이 높다. 원시 파일은 감사·비교용으로 보관하되, 새 연구는 별도의 `research_v2` 데이터 계보에서 시작해야 한다. 기존 앱 코드는 범용 엔진과 UI 컴포넌트만 선별해 재사용하고, 연구 데이터와 결과 수치는 새 프로토콜에서 다시 생성한다.
